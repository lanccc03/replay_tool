Implementation Approaches

Two ways to structure the change, with trade-offs

A

QStackedWidget inside ScenariosView (Recommended)

Replace the current QSplitter with a QStackedWidget that holds two pages: list_view (toolbar + table) and editor_view (flat editor with all sections). The editor is built directly in ScenariosView. This keeps changes contained to one file and follows the existing pattern.

Pros

  • Minimal structural changes — stays in one file
  • Matches the "replace in place" UX exactly
  • ViewModel already supports draft editing

Cons

  • scenarios_view.py is already 1915 lines — grows larger
  • Editor logic mixed with list logic
B

Separate ScenarioEditorView class

Extract the editor into its own view file (scenario_editor_view.py). ScenariosView uses QStackedWidget to show either a read-only list page or the separate editor widget. Cleaner separation but involves more new files.

Pros

  • Smaller, focused files — editor is 300-400 lines not bundled
  • Easier to test independently
  • Better follows MVVM separation

Cons

  • More files to create and wire up
  • Slightly more boilerplate
